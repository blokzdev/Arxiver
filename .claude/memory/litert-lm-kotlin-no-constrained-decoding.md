name: litert-lm-kotlin-no-constrained-decoding
description: LiteRT-LM's Kotlin binding can't do constrained/grammar decoding (#1662); PA.4 is "app-draws-the-structure", not constrained decoding
type: gotcha
---

PA.0c finding (2026-06-26, primary-source verified). LiteRT-LM's **C++** API supports constrained decoding via LLGuidance (`LlGuidanceConstraintArg` with JSON-schema / Lark / regex), but the **Kotlin/Android binding does NOT surface it**: `SamplerConfig` exposes only `topK/topP/temperature/seed`; `EngineConfig`/`SessionConfig`/`ConversationConfig` carry no constraint/schema/grammar field; the only Kotlin lever is the experimental boolean `ExperimentalFlags.enableConversationConstrainedDecoding`, which constrains **tool-call argument syntax only** (not an arbitrary user schema). Tracked upstream as **open issue google-ai-edge/LiteRT-LM#1662** (opened 2026-03-18, no maintainer timeline); the C++ constrained stack keeps expanding (NPU/GPU PRs) which *widens* the Kotlin gap. Don't plan any feature on Kotlin constrained decoding; if it ever lands, #1662 is the unblock signal.

**Decision that follows (PA.4):** we rejected constrained decoding as the **wrong tool**, not just unavailable. At our model scale it trades content for syntax — "Constraint Tax" (2026, on Qwen2.5-0.5B/1.5B) measured structural validity →100% but accuracy 19.7%→11.0% and wrong-but-valid 49.5%→88.9%; "Let Me Speak Freely?" (arXiv:2408.02442), CRANE, DCCD all say decouple reasoning from formatting. So **PA.4 = "app-draws-the-structure"**: the model emits a low-syntax sentinel-delimited intermediate (small models hit a simple delimited format near-perfectly 0-shot — arXiv:2506.19512 — for less prompt budget than a table exemplar), a pure-Kotlin tolerant parser builds a typed `ComparisonTable`, and a deterministic renderer emits guaranteed-valid GFM/Compose with a confidence-gated bulleted-list fallback (the ~1B "table reversal", arXiv:2412.17189). Invalid tables become structurally impossible — and it extends PA.1's deterministic-artifact thesis. Also rejected: a 2nd runtime (llama.cpp/MLC/ONNX-GenAI all fail ≥1 of {native-Kotlin, CPU, sub-2B, no-JNI}) and a custom JNI LLGuidance shim (buys syntax the app already guarantees). Model-side lever if a tier ever *must* emit structure = a tiny fine-tune (FunctionGemma-270M 58%→85%), never runtime constraints. See [[gemma-structured-table-validity]] and [[qwen-light-tier-modelspec]].
